# API 工作流程详细说明

TikTok API 工作流程采用 **Design-First** 方法，将 API 描述作为单一事实来源，提高 API 质量。Design-First 意味着团队中的每个人都在使用相同的语言，采用这种方法可以帮助企业和团队弥合业务与技术之间的差距。

## 工作流程阶段

API 工作流程包含四个主要阶段：

### 1. 设计阶段
从需求构建 API 规范，产出物包括 API 规范文档和生成的 API 文档。

**设计流程**：
1. 评审需求文档，理解需求
2. 探索现有 API 以确保命名一致性
3. [可选] 创建服务（包含多个 API），指定描述、安全配置等元信息
4. 创建数据模型，添加示例
5. 创建操作（端点）
   - 设置请求和响应，尽可能重用数据模型
   - 创建/重用公共组件（头部、参数、示例、响应）
   - 添加错误响应
6. 提交更改进行评审（自动生成文档/预览/Mock）

### 2. 原型阶段
基于 API 规范创建 Mock API 以验证和评审设计。

**Mock 方法**：
- **静态 Mock**：使用手动数据的 Mock 服务器
  - 本地工具：AnywhereDoor、Charles
  - 公共 Mock 服务器工具：BAM、ByteMock
  - 外部工具：Postman
- **动态 Mock**：使用代码生成动态数据的 Mock 服务器

**验证要点**：
- 接口方法（方法名、返回类型、参数长度和类型）
- 字段（缺失字段、类型不匹配、名称不匹配）
- 语法（语法错误、未定义符号、基本约束）
- HTTP 响应码（2xx/3xx/4xx/5xx）
- HTTP 头部（Content-Type、Connection、Authorization、X-Forwarded-For）

### 3. 开发与测试阶段
API 生产者和消费者并行开发，基于最终化的契约进行实现。

**开发流程**：

**代码生成策略**：
- **公共结构代码生成**：通过自动化工具生成，在 API 设计发布时触发，存储到集中式代码库
- **服务端骨架代码生成**：提供骨架代码最佳实践，开发者通过生成工具生成 API 骨架代码
- **客户端代码生成**：通过自动化工具生成，在 API 设计发布时触发

**测试类型**：
- **CI/CD 自动化测试**：API 单元测试、性能测试、差异测试
- **环境测试**：测试环境（BOE）、预生产环境（PPE）

### 4. 消费阶段
消费者发现和集成现有 API。

## API 类型分类

根据核心消费者和使用范围，API 分为五种类型：

| 类型 | 描述 |
|------|------|
| **Public** | 开放给外部开发者或业务使用，API 本身作为产品 |
| **Partner** | 仅对特定授权的外部开发者可用 |
| **Internal** | 仅在公司内部使用，连接系统和数据 |
| **Private** | 在 IDC 外部不可访问，如 RPC 服务 |
| **Composite** | 仅组合来自不同 API 的数据，如 BFF 层提供的 API |

## 文档编写

在 API 开发过程中，技术作者或 API 生产者应开始完善 API 文档。

**文档结构**：
- **介绍**：描述 API 提供的功能、限制、核心概念
- **快速入门**：先决条件、集成步骤、验证集成的示例、故障排除说明
- **API 参考**：所有 API 列表及简单描述，以及详细信息（模型、错误代码等）和每个端点的示例
- **教程和操作指南**：如何完成常见任务

## 发布管理

API 生产者发布 API 时，必须将最新信息发布到 API 目录。API 目录应按类别组织（如 `TikTok/Comment`），并包含工作流程早期产出的链接。

**Design-First 工作流程的优势**：
- **以客户为中心的 API**：产品经理和业务领导更早参与，帮助塑造 API 功能
- **设计驱动开发**：API 开发者可以使用 API 设计来驱动开发工作
- **快速反馈**：通过尝试 Mock API 及其文档来评审和验证设计
- **并行开发**：消费者可以在 API 完成前开始构建 API 客户端
- **重用现有 API 和模型**：在设计阶段投入思考和时间，可以探索现有组合中的可重用设计和最佳实践
- **一致性**：自动样式指南和现有 API 设计确保整个组织的 API 提供一致的开发者体验
