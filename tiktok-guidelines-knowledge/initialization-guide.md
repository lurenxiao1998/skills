# 初始化规范指南

本文档详细说明TikTok Go代码规范中的初始化相关要求，确保代码的一致性和可读性。

## 常量初始化

### 编译时限制
- **必须**：常量表达式必须是编译器可评估的常量表达式，在编译时创建
- **特性**：常量在Go中是恒定的，即使定义为函数内的局部变量，也只能是数字、字符（rune）、字符串或布尔值

## 变量初始化

### 运行时初始化
- **灵活性**：变量可以像常量一样初始化，但初始化器可以是运行时计算的一般表达式
- **时机**：变量初始化在运行时进行，与常量的编译时初始化形成对比

## 结构体初始化

### 字段命名要求
- **应该**：初始化结构体时几乎总是指定字段名称，现在由`go vet`强制执行

**不推荐示例**：
```go
k := User{"John", "Doe", true}
```

**推荐示例**：
```go
k := User{
    FirstName: "John",
    LastName: "Doe",
    Admin: true,
}
```

### 空结构体声明
- **应该**：当结构体的所有字段在声明中被省略时，使用`var`形式声明结构体

**不推荐示例**：
```go
user := User{}
```

**推荐示例**：
```go
var user User
```

## 结构体引用初始化

### 初始化方式
- **应该**：使用`&T{}`而不是`new(T)`初始化结构体引用，以保持与结构体初始化的一致性

**不推荐示例**：
```go
// 不一致
sval := T{Name: "foo"}
sptr := new(T)
sptr.Name = "bar"
```

**推荐示例**：
```go
sval := T{Name: "foo"}
sptr := &T{Name: "bar"}
```

## 映射初始化

### 空映射初始化
- **应该**：对空映射和程序化填充的映射使用`make(..)`
- **优势**：这使得映射初始化在视觉上与声明不同，并且便于在可用时添加大小提示

**不推荐示例**：
```go
var (
    // m1可以安全读写
    // m2在写入时会panic
    m1 = map[T1]T2{}
    m2 map[T1]T2
)
// 声明和初始化在视觉上相似
```

**推荐示例**：
```go
var (
    // m1可以安全读写
    // m2在写入时会panic
    m1 = make(map[T1]T2)
    m2 map[T1]T2
)
// 声明和初始化在视觉上不同
```

## init函数使用

### 避免原则
- **应该**：尽可能避免使用`init()`函数
- **理由**：`init()`函数在包的所有变量声明评估其初始化器之后调用，这些初始化器仅在所有导入的包初始化后才进行评估

### 使用建议
当`init()`不可避免或可取时，代码应尝试：

1. **避免依赖**：避免依赖其他`init()`函数的顺序或副作用。虽然`init()`顺序是众所周知的，但代码可以更改，因此`init()`函数之间的关系可能使代码脆弱且容易出错

2. **避免全局状态**：避免访问或操作全局或环境状态，如机器信息、环境变量、工作目录、程序参数/输入等

3. **避免I/O**：避免I/O，包括文件系统、网络和系统调用

**不推荐示例**：
```go
type Foo struct {
    // ...
}
var _defaultFoo Foo

func init() {
    _defaultFoo = Foo{
        // ...
    }
}
```

**推荐示例**：
```go
var _defaultFoo = Foo{
    // ...
}

func defaultFoo() Foo {
    return Foo{
        // ...
    }
}
```

## 关键要点总结

1. **常量与变量**：区分编译时常量和运行时变量的初始化要求
2. **结构体初始化**：始终使用字段名称以提高代码清晰度
3. **引用初始化**：使用`&T{}`语法保持一致性
4. **映射初始化**：使用`make()`进行显式初始化
5. **init函数**：尽可能避免使用，优先选择显式初始化函数

这些规范确保TikTok代码库中的初始化代码具有一致的模式，提高代码的可读性和可维护性。
