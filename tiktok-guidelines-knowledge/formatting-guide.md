# 格式规范指南

TikTok代码规范中的格式规范部分主要关注代码的可读性和一致性，确保所有TikTok工程师遵循统一的编码风格。

## 格式化工具要求

**必须**依赖`gofmt`和`goimports`工具来保持与业界标准的一致性。这些工具的使用是强制性的，确保代码格式符合Go语言的官方标准。

## 导入分组规范

导入语句的组织需要遵循特定的分组模式，以提高代码的可读性：

### 基本分组原则
导入应组织成两个基本组：标准库包和其他包。可以接受将第二组进一步细分为多个子组，但这需要作者和评审者共同维护。

### 推荐的分组模式
TikTok推荐以下四种分组模式中的任意一种：

1. **两组模式**：
   - 标准库包
   - 其他包

2. **三组模式**：
   - 标准库包
   - 依赖包
   - 项目包

3. **三组模式（最推荐）**：
   - 标准库包
   - 外部包（不以'code.byted'开头）
   - 内部包

4. **四组模式**：
   - 标准库包
   - 外部包
   - 内部依赖包
   - 项目包

## 行长度处理

对于行长度，没有硬性限制。但**应该**在提高可读性时换行，而不仅仅因为行长度。这意味着换行决策应基于代码的可读性，而不是机械地遵循某个字符数限制。

## 相关检查规则

TikTok使用golintx规则来强制执行格式规范，包括：
- `GO-520-5UN37V byted_s_import_group_any` - 导入分组检查规则

## 实践建议

1. **一致性优先**：在整个项目中保持一致的导入分组模式
2. **工具自动化**：充分利用`goimports`自动整理导入语句
3. **可读性导向**：以代码可读性为最高原则，而不是死板地遵循规则
4. **团队协作**：在团队内统一格式规范，减少代码审查时的格式争议

## 示例代码

```go
// 推荐的三组模式示例
import (
    // 标准库包
    "fmt"
    "os"
    "time"
    
    // 外部包
    "github.com/gin-gonic/gin"
    "gorm.io/gorm"
    
    // 内部包
    "code.byted.org/tiktok/common"
    "code.byted.org/tiktok/user"
)
```

通过遵循这些格式规范，可以确保TikTok代码库中的代码具有一致的外观和良好的可读性，从而提高团队协作效率和代码维护性。
