# ByteCheck规则集详细说明

ByteCheck是TikTok内部的代码规范检查工具，用于自动化检查代码是否符合TikTok编码规范。规则集涵盖多个编程语言，每个规则都有特定的编号、等级和描述。

## Go语言规则

### 函数注释规则
- **规则编号**: GO-520-DCA0EF
- **规则等级**: info
- **规则名称**: byted_s_func_comment_except_override
- **规则描述**: 导出的函数必须有注释描述其用法
- **注释说明**: 如果函数名称已经足够清晰，则不需要注释

### 函数嵌套深度规则
- **规则编号**: GO-520-M03NFB
- **规则等级**: error
- **规则名称**: byted_s_too_many_nests_in_func
- **规则描述**: 函数内嵌套块深度不能超过3层
- **修订理由**: 函数复杂性约束，嵌套深度≤3（参考Linus Torvalds的观点：如果需要超过3层缩进，程序就有问题，应该修复）

### 函数参数数量规则
- **规则编号**: GO-520-CCA1CC
- **规则等级**: warning
- **规则名称**: byted_s_args_length_limit
- **规则描述**: 单个函数的参数不应超过5个
- **注释说明**: 不超过8个参数
- **修订理由**: 函数复杂性约束，函数参数>5显示warning，>8显示error

### 循环嵌套规则
- **规则编号**: GO-520-MT3FOR
- **规则等级**: error
- **规则名称**: byted_s_too_many_for_nest
- **规则描述**: for语句嵌套深度不能超过3层
- **状态**: 已关闭，由byted_s_too_many_nests_in_func规则统一约束

### 条件嵌套规则
- **规则编号**: GO-520-M03XIF
- **规则等级**: error
- **规则名称**: byted_s_too_many_if_nests
- **规则描述**: if语句嵌套块深度不能超过3层
- **状态**: 已关闭，由byted_s_too_many_nests_in_func规则统一约束

### 结构体字节数组规则
- **规则编号**: GO-520-D9F1F3
- **规则等级**: warning
- **规则名称**: byted_s_struct_byte_array
- **规则描述**: 结构体不应使用[]byte类型，可以序列化为base64
- **状态**: 已关闭，[]byte类型json marshal成字符串是正常行为

### 错误比较规则
- **规则编号**: GO-520-A4F864
- **规则等级**: warning
- **规则名称**: byted_s_error_binary
- **规则描述**: 错误类型标识应使用errors.Is判断，而不是==或!=
- **TikTok新规范**: error级别
- **修订理由**: 防御型规范，错误比较应使用errors.Is，不要用==

### 自定义错误定义规则
- **规则编号**: GO-520-CF3543
- **规则等级**: warning
- **规则名称**: byted_s_error_define
- **规则描述**: 自定义错误类型应有Unwrap()方法
- **TikTok新规范**: info级别
- **修订理由**: 当错误比较复杂且有着共同的上下文信息时，可以考虑引入自定义错误，此时自定义错误需要放在error.go文件中。自定义错误类型的命名应该以Error结尾。除了Error()，还要加上Unwrap()方法，用于返回其结构体中的error变量。如果这个结构体只用于初次产生error，里面不会wrap其它error就不需要

## 规则等级说明

### error级别
- 表示严重违反规范，必须修复
- 示例：函数嵌套深度超过3层

### warning级别
- 表示建议改进，但不是必须
- 示例：函数参数超过5个

### info级别
- 表示信息性提示，供参考
- 示例：导出的函数缺少注释

## 规则修订流程

1. **修订理由**：每个规则都有明确的修订理由，说明为什么需要该规则
2. **存量修复成本评估**：
   - 容易：可以脚本修复或情况较少
   - 中等：需要手动改，但很好改且不会产生问题
   - 困难：改动有些复杂或容易引起问题
3. **建议处理**：包括取消、提升或降低等级、排除特殊情况等

## 使用建议

1. **集成到CI/CD**：将ByteCheck规则集成到持续集成流程中
2. **本地开发检查**：在提交代码前使用ByteCheck进行本地检查
3. **代码审查参考**：作为代码审查的客观标准
4. **团队规范统一**：确保跨团队代码风格一致性

## 注意事项

- 部分规则可能已关闭，由更通用的规则替代
- 规则等级可能根据TikTok当前规范和新规范进行调整
- 自定义规则需要明确修订理由和修复成本评估
