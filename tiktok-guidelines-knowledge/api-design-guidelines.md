# API 设计指南详细规范

本文档详细说明 TikTok API 设计的具体规范和最佳实践，为 API 设计提供全面的编程指导。

## 设计原则

### API First, Design First
采用设计优先的方法，在实现之前创建可读的 API 规范，确保设计质量和早期反馈。

### 资源导向设计模式
- HTTP API 设计**应该**使用资源导向设计模式
- RPC API 设计**推荐**尽可能使用资源导向设计模式
- 资源导向 API **应该**建模为资源层次结构，每个节点是简单资源或资源集合
- 资源导向 API **必须**在无状态协议上运行

### 隐藏复杂性
- API **应该**设计得让常见用例尽可能简单
- 简单的接口比简单的实现更重要

### 保持一致性
确保 API 在签名、命名、文档和运行时行为上保持一致。

### 错误预防
尽可能通过设计消除错误，而不是依赖错误处理。

## 命名规范

### 资源命名
- 所有资源名称在 API 内**必须**唯一
- 资源 ID **不应该**使用大写字母
- 集合标识符**必须**是简洁的美式英语术语，**必须**是资源名词的复数形式
- 集合标识符**必须**以小写字母开头，仅包含 ASCII 字母和数字

### 重复字段
- 重复字段**必须**使用复数字段名
- 重复字段**应该**有强制上限，建议不超过 100 个元素
- 重复字段**不能**内联表示另一个资源的主体

### 自定义方法
- RPC 方法名：**应该**形如"动词+名词"，**不能**包含介词
- HTTP 动词：**应该**通常使用 POST；如果是 get 或 list 方法的替代，**应该**使用 GET
- HTTP URI **必须**是冒号后跟自定义动词，URI 中的动词**必须**与方法名中的动词匹配

### 字段命名
- Protobuf 文件中的字段定义**必须**使用 `lower_snake_case` 命名
- Thrift 文件中的字段定义**必须**使用 `CamelCase` 命名
- 字段名**不能**以数字开头，**不能**包含前导、尾随或相邻下划线
- 字段名**应该**避免使用介词
- 包含形容词和名词的字段名**应该**将形容词放在名词前
- 布尔字段**应该**省略"is"前缀
- 字段名**应该**避免与常见编程语言的关键字冲突

### 枚举
- 所有枚举值**必须**使用 `UPPER_SNAKE_CASE`
- 枚举的第一个值**应该**是枚举名后跟 `_UNSPECIFIED` 后缀
- 其他值**不应该**以枚举名本身为前缀
- 仅在单个消息中使用的枚举**应该**嵌套在该消息内
- 枚举**应该**接收新值不频繁

## 方法定义

### 标准方法
API **应该**优先使用标准方法而非自定义方法。

| 标准方法 | HTTP 动词 | RPC 方法名 |
|---------|-----------|------------|
| Get | GET | Get + 资源消息名单数形式 |
| List | GET | List + 被列出资源的复数形式 |
| Create | POST | Create + 被创建资源的单数形式 |
| Update | PATCH/PUT | Update + 资源消息名单数形式 |
| Delete | DELETE | Delete + 资源消息名单数形式 |

### 请求和响应消息
- 请求消息**必须**匹配方法/RPC 名，带有 **Request** 后缀
- Get/Create/Update 方法：响应消息**必须**是资源本身
- List 方法：响应消息**必须**匹配方法/RPC 名，带有 **Response** 后缀
- Delete 方法：响应消息**应该**是默认的 Empty 结构

### 唯一标识符
- **应该**有唯一标识符字段用于单个资源实例操作
- 优先使用 64 位整数或唯一字符串作为标识符字段，名称为 `ID`

### 方法可见性
- RPC API **应该**默认私有，仅在必要时公开
- 此规则不适用于 HTTP API 设计

## 文档规范

### 通用指导
- 公共注释**必须**包含在几乎所有组件上
- 注释**应该**使用语法正确的美式英语
- 每个注释的第一句话**应该**省略主语，使用第三人称现在时

### 服务和方法的文档
- **应该**先写注释，因为注释是捕获抽象的唯一方式
- 注释**应该**以描述方法行为的一两句话开始
- 如果 API 有任何副作用，这些**必须**被记录
- 注释**必须**描述方法可能产生的任何异常/错误
- 必须描述调用方法前必须满足的任何前提条件

### 字段文档
- 注释**应该**使用与字段名不同的词语
- 注释**应该**填写缺失的细节，如单位、空值的含义、不变量等

### 格式化和链接
- 注释**应该**遵循约定，注释中的任何格式化**必须**使用 CommonMark/Markdown 语法
- 内部链接**应该**使用元素的完全限定名作为 Markdown 引用链接
- 外部链接**必须**使用绝对 URL

### 弃用
- 要弃用服务/消息/字段，**必须**设置弃用选项
- 相应注释的第一行**必须**以 `Deprecated: ` 开头并提供替代方案

## 目录和文件结构

### 目录结构
- API 设计文件**必须**放在公共仓库中并用 git 管理
- TikTok HTTP API IDL：https://code.byted.org/tiktok/idl
- TikTok Live HTTP API IDL：https://code.byted.org/tikcast/api_idl/

### HTTP API IDL 目录布局
```protobuf
repo(tiktok/idl)
├── common
│   ├── common.proto
│   └── social_common.proto
├── api
│   ├── social value chain
│   │   └── aweme_v1_comment_publish.proto
│   └── content discovery value chain
├── service
│   ├── tiktok_comment_api_service.proto
│   └── tiktok_video_consumption_api_service.proto
```

### 文件规范
- 文件名称**必须**使用 `snake_case`
- HTTP API **应该**在 Protocol Buffers 中定义
- RPC API **必须**在 Thrift 协议中定义
- 单个文件**应该**将更高级和更重要的定义放在较低级和较不重要的定义之前

## HTTP 特定规范

### 页面令牌
对于列表操作，**应该**添加页面令牌和下一页令牌，如果包含潜在敏感信息则进行加密。

### 响应格式
定义 **Response** 时，**必须**在每个 HTTP 响应中使用 status_code/status_msg/extra 字段：

```json
{
   "data": {
       ...
   },  
   "status_code": ..,  
   "status_msg": ..,
   "extra": {
      "now": .., 
      "logid": ..,
   }
}
```

## RPC 特定规范

### Base 和 BaseResp
定义 **Response** 时，**必须**在每个请求中使用 `Base` 字段，在每个响应中使用 `BaseResp` 字段。

## 错误处理

### 错误代码
- API 的业务错误代码**应该**在一个统一的仓库/文件中定义和分配
- API 的错误代码**必须**区分底层基础设施错误和业务错误代码

### 错误消息
- 错误消息**不应该**假设用户是特定 API 的专家
- 错误消息**应该**简洁但可操作
- 错误消息**必须**使用英文

### 部分错误
API **不应该**支持部分错误，这会为用户增加显著的复杂性。

### 客户端重试
- 具有自动客户端重试逻辑的客户端库**应该**提供一种机制，让用户指定要重试的错误代码和重试延迟
- 客户端**不应该**自动重试事务性请求或重复运行会导致意外状态变化的请求

## 设计模式

### 分页
- 返回数据集合的 API **必须**从一开始就支持分页
- 请求消息**应该**定义 `page_size` 字段
- 请求消息**应该**定义字符串 `page_token` 字段
- 响应消息**应该**定义字符串 `next_page_token` 字段
- API 提供的页面令牌**必须**是不透明的（但 URL 安全的）字符串，**不能**是用户可解析的

### 过滤
- 基本过滤：为 List 操作添加过滤器字段以支持默认的 `equal+and` 过滤策略
- 高级过滤：基于特定需求支持高级过滤策略，包括比较运算符、逻辑运算符和包含运算符

## 安全规范

### 隐私和合规
- 接口**应该**根据不同的使用环境进行隔离
- 如果 API 涉及敏感数据，此接口设计**必须**在发布前经过合规评估
- 用户敏感数据的任何外部披露和显示**必须**进行脱敏
- 用户的高度敏感个人信息**必须**被**禁止**以任何方式传输或备份

### 反爬虫/反作弊/反窃听
- **必须**禁止使用可遍历字段作为用户的外部标识符
- 登录/注册页面**必须**连接到验证码系统或其他频率控制机制
- 敏感信息**必须**使用加密通道传输
- 所有 API **推荐**接入反爬虫中间件或启用 Janus/AsgwLoader

### 认证和授权
- 对于用户数据或敏感数据 API **必须**有认证策略来识别请求所有者
- 进行授权时，**应该**遵循最小权限原则
- 认证失败时，**应该**返回众所周知的权限拒绝错误 `PERMISSION_DENIED`

### HTTP 特定安全
使用 ETag 保护删除操作，当使用 ETag 删除资源时，**必须**接受并与服务器计算的 ETag 进行检查。

### RPC 特定安全
- RPC 服务**应该**使用 GDPR/ZTI 和 ACL 设施来认证和授权所有请求调用者
- RPC 服务**应该**使用**最终用户票据**（ByteTIM）来执行敏感场景的认证和授权

## API 设计评审

### 评审范围
- 几乎 TikTok-ENG 范围内使用的知名 Server API（RPC）**应该**被评审
- TikTok 移动应用/Web 应用使用的所有客户端 HTTP API **应该**被评审
- 向其他团队公开的 Server API（RPC）**高度推荐**被评审

### 角色和责任
- **API MR 所有者**：设计和构建 API 的利益相关者
- **设计评审者**：进行评审并提供指导的经验丰富的 API 设计评审者
- **影子设计评审者**：正在培训中的设计评审者

### 评审流程
1. API MR 所有者请求对其 API 规范进行评审，并被分配一个设计评审者
2. 设计评审者根据设计指南异步评审 API 规范，并直接向 API MR 所有者提供反馈
3. 评审来回进行，直到设计评审者批准 API MR
